    Sprint Notes — 2026-03-13

    Status: Planning.

    Goal

    Same end goal as the 2026-03-12 sprint — make the current architecture runnable end-to-end — but with PR K and onward redesigned around the agent communication protocol defined in docs/AGENT_COMM_PROTOCOL.md.

    Context

    The 2026-03-12 sprint shipped PR J (ops/ infrastructure primitives) and began work on PR K (instruction builders). During PR K development, we identified that the instruction builders were porting the prototype's monolithic approach — mixing role-specific work with git commands, signaling logic, and framework-specific actions in a single text blob.

    We paused to design a clean communication protocol from first principles (PR #91, docs/AGENT_COMM_PROTOCOL.md). The protocol defines five layers:

    1. Task manifest (manifest.json) — structured facts about the task
    2. Work instructions (instructions.md) — role-specific work only, template-driven for formulaic roles
    3. Workflow policies (policies.json) — composable JSON, abstract vocabulary that adapters translate
    4. Signaling contract — abstract signal definitions (done, failed, concern, gate_attempt)
    5. Framework adapter — maps onto existing TaskRunnerIO protocol implementations

    This sprint picks up from where 2026-03-12 left off, with the protocol as the foundation for all remaining work.

    What the protocol changes

    • PR K is no longer "port prototype instruction builders." It's now "build the framework-agnostic protocol layers (manifest, policies, templates, spec representation)."
    • PR L's TaskPreparer now writes manifest.json + policies.json + resolved instructions.md (via template or custom), not monolithic instruction text.
    • PRs M–P are mostly unchanged.

    PR plan

    PR K: Protocol schemas, builders, and templates

    • Branch: feat/protocol-schemas

    Framework-agnostic data types and builders for protocol layers 1-3. No shell commands, no framework-specific data anywhere in this PR.

    Modules:

    Module Responsibility
    protocol/manifest.py TaskManifest dataclass + build_manifest(task, runtime) → dict
    protocol/policies.py WorkflowPolicies dataclass + build_policies(task, runtime) → dict
    protocol/templates.py resolve_instructions(role, manifest, adapter_name?) → str — loads and parameterizes role templates
    spec/ package SpecRepresentation protocol + PythonStubSpec first impl

    Templates directory: src/agentrelay/templates/

    Shared role templates as markdown files:

    File Role
    spec_writer.md Creative spec writing — parameterized by spec representation type
    test_writer.md Write tests against specs — parameterized by paths
    test_reviewer.md Review tests — parameterized by paths
    implementer.md Implement code, pass tests — parameterized by paths + gate

    Adapter-specific overrides go in templates/<adapter_name>/ (e.g., templates/claude_code/test_writer.md). Resolution order: adapter-specific > shared > per-task custom.

    Design notes: - Manifest builder extracts data from Task + TaskRuntime and produces the JSON structure defined in AGENT_COMM_PROTOCOL.md Layer 1. - Policies builder reads Task.completion_gate, Task.review, etc. and produces the composable JSON structure from Layer 3. - Template resolver uses Task.role to select template, substitutes {src_paths}, {test_paths}, etc. from manifest data. Falls back to per-task instructions.md for generic role. - SpecRepresentation protocol defines how spec formats are described to agents. PythonStubSpec = Python files with docstrings and raise NotImplementedError.

    Acceptance criteria: - [ ] Manifest builder produces valid JSON matching the protocol schema. - [ ] Policies builder produces valid JSON matching the protocol schema. - [ ] Template resolver loads templates, substitutes parameters correctly. - [ ] generic role falls back to custom instructions. - [ ] SpecRepresentation protocol and PythonStubSpec are defined. - [ ] pixi run check passes.

    PR L: Task-level protocol implementations

    • Branch: feat/task-protocol-impls
    • Package: src/agentrelay/task_runner/implementations/

    Concrete classes implementing all six TaskRunnerIO protocols. Each class composes ops/ primitives and protocol builders from PR K.

    Class Satisfies Key operations
    WorktreeTaskPreparer TaskPreparer git.branch_create(), git.checkout() (in shared workstream worktree), write manifest.json (via manifest builder), write policies.json (via policies builder), write instructions.md (via template resolver), optionally write context.md
    TmuxTaskLauncher TaskLauncher tmux.new_window(), launch Claude Code, return TmuxAgent
    TmuxTaskKickoff TaskKickoff Delegate to agent.send_kickoff()
    SignalCompletionChecker TaskCompletionChecker signals.poll_signal_files(), parse into TaskCompletionSignal
    GhTaskMerger TaskMerger gh.pr_merge(), git.pull_ff_only()
    WorktreeTaskTeardown TaskTeardown tmux.capture_pane(), tmux.kill_window(), git.branch_delete()

    Also in this PR: Complete TmuxAgent stubs (from_config() and send_kickoff()) using ops/tmux.py.

    Acceptance criteria: - [ ] Each class satisfies its protocol (isinstance check). - [ ] Unit tests with mocked ops/ functions cover success and failure paths. - [ ] TaskPreparer writes all three protocol files to the signal directory. - [ ] TaskRunnerIO can be constructed from all six implementations. - [ ] pixi run check passes.

    Resolved questions

    Carried forward from 2026-03-12 where still applicable:

    1. Package name for protocol implementations. Resolved in PR #88: each package uses core/ + implementations/ subpackages.

    2. Instruction builder scope. Superseded: PR K now builds protocol layers 1-3 per AGENT_COMM_PROTOCOL.md. Role-specific instructions use templates; no shell commands or signaling in instruction content.

    3. WorkstreamRunner integration into Orchestrator. Separate PR (N½).

    4. ops/ testing strategy. Layered approach (real git repos, subprocess mocks for tmux/gh, real filesystem for signals).

    5. Where do role templates live? src/agentrelay/templates/ with adapter-specific overrides in templates/<adapter_name>/.

    6. Protocol file format. Per AGENT_COMM_PROTOCOL.md: manifest.json (Layer 1), instructions.md (Layer 2), policies.json (Layer 3).
