## Goal

Systematically build out end-to-end test coverage by category. Each category
corresponds to a subdirectory under `graphs/` with YAML graphs, README docs,
and observations from test runs. Fix bugs and wire unimplemented features as
needed to enable each test category.

outcome reporting bug in PR #113.

written to policies.json) but not wired in the runner: completion gates, review,
and ADR production. A concerns bug was also found (signal.concerns dropped in
the runner).

## PR plan

### PR A: Agent-side helper + fix concerns bug + concerns/ test graphs — **Done** (PR #114)

- Branch: `fix/concerns-capture`

**Agent-side helper:**
Create an agent-side helper class that agents instantiate inside their tmux
session to interact with the orchestrator via a clean Python API. Replaces
raw shell commands (`echo ... > .done`) with method calls
(`helper.mark_done(pr_url)`). Reads task metadata from `manifest.json` in
`$AGENTRELAY_SIGNAL_DIR`.

Methods:
- `mark_done(pr_url)` — write .done signal
- `mark_failed(reason)` — write .failed signal
- `record_concern(concern)` — append to concerns.log
- Class method `from_env()` — instantiate from `$AGENTRELAY_SIGNAL_DIR`

Update the workflow footer in `agent_comm_protocol/templates.py` to instruct
agents to use the helper API instead of raw shell commands. Include concern
recording guidance for all roles (optional — "if you have concerns, record
them").

**Bug fix (concerns transfer):**
- `StandardTaskRunner.run()` reads `signal.concerns` from the completion
  checker but never transfers them to `runtime.artifacts.concerns`. Add the
  missing assignment. (Already implemented on `fix/concerns-capture` branch.)

**Test graphs:**
- `graphs/concerns/README.md` — category docs
- Graph with a task whose description instructs the agent to record concerns
  via the helper API. Verify they appear in the orchestrator result.

**Acceptance criteria:**
- [x] Agent-side helper class works when imported by agent in tmux session
- [x] Workflow footer uses helper API for completion signaling and concerns
- [x] Concerns from `concerns.log` are stored in `runtime.artifacts.concerns`
- [x] Concerns appear in console output
- [x] `pixi run check` passes
- [x] E2E: concerns graph runs, agent uses helper, concerns are captured

### PR D: Wire completion gates + gates/ test graphs

- Branch: `feat/completion-gates`

**Wire gate execution:**
- `TaskGateChecker` protocol + `GateCheckResult` in `task_runner/core/io.py`
- `ShellGateChecker` implementation: runs shell command in worktree, captures
  output to `gate_last_output.txt`, returns pass/fail
- Gate step in `StandardTaskRunner.run()`: after completion signal + PR_CREATED,
  before merge. Retry loop up to `max_gate_attempts` (default 5)
- Gate failure marks task FAILED with `failure_class=None` (retryable by
  orchestrator when `max_task_attempts` allows)
- Events: `task_gate_running` (verbose), `task_gate_passed` (verbose),
  `task_gate_failed` (always visible)
- Direct field on runner (not StepDispatch) — gate doesn't vary by
  framework/environment

**Test graphs:**
- `graphs/gates/` category with `pixi_run_test.yaml` (gate-pass scenario)
- `graphs/failure/retry_on_gate_failure.yaml` (gate-triggered retry with
  `--max-task-attempts 2`)

**Acceptance criteria:**
- [x] Gate command runs after task signals .done
- [x] Gate failure triggers retry (up to max_gate_attempts)
- [x] Gate pass proceeds to PR merge
- [ ] Retry: agent's second attempt passes the gate after first attempt fails
- [x] Verbose output shows gate events
- [x] `pixi run check` passes (957 tests)
