# Sprint Notes — 2026-03-12

> **Status: Planning.**

## Goal

Implement concrete classes that satisfy the protocol boundaries already defined
in `task_runner/io.py` and `workstream/io.py`, complete the `TmuxAgent` stubs,
and provide a CLI entry point — making the current architecture runnable
end-to-end.

## Context

The current architecture defines a complete scheduling and lifecycle layer:

- **`Orchestrator`** schedules tasks via dependency + workstream constraints,
  delegating one-task execution to anything satisfying `TaskRunnerLike`.
- **`TaskRunner`** drives the `PENDING → RUNNING → PR_CREATED → PR_MERGED`
  state machine, delegating each lifecycle step to the six per-step protocols
  composed in **`TaskRunnerIO`**.
- **`WorkstreamRunner`** drives workstream prepare/merge/teardown, delegating
  to the three per-step protocols in **`WorkstreamRunnerIO`**.

All of this is tested (~624 tests) against in-memory fakes. What's missing are
**concrete implementations of the nine per-step protocols** and a CLI that wires
everything together.

### Protocol → concrete operation mapping

Each protocol maps to a small set of subprocess operations already proven in the
v01 prototype (`prototypes/v01/task_launcher.py`, `run_graph.py`):

| Protocol | What the implementation does | Infrastructure needed |
|---|---|---|
| **`TaskPreparer`** | Create git worktree + branch, create signal dir, write `task_context.json` and `instructions.md` | git worktree ops, signal file I/O, prompt/instruction builders |
| **`TaskLauncher`** | Create tmux window, launch Claude Code in it, return `TmuxAgent` | tmux ops, `TmuxAgent.from_config()` |
| **`TaskKickoff`** | Wait for Claude TUI readiness, send kickoff prompt via tmux | tmux ops, `TmuxAgent.send_kickoff()` |
| **`TaskCompletionChecker`** | Poll signal dir for `.done` / `.failed`, parse into `TaskCompletionSignal` | signal file I/O |
| **`TaskMerger`** | `gh pr merge`, pull main / graph branch to sync local refs | gh CLI ops, git fetch/pull |
| **`TaskTeardown`** | Capture agent log, close tmux pane, remove worktree + branch | tmux ops, git worktree ops |
| **`WorkstreamPreparer`** | Create `graph/<name>` integration branch, push to origin | git branch ops |
| **`WorkstreamMerger`** | Create PR from graph branch → main, merge it | gh CLI ops |
| **`WorkstreamTeardown`** | Delete graph branch (local + remote) | git branch ops |

### What this means for PR structure

The previous sprint plan (2026-03-09) organized PRs around "adapters" — one per
infrastructure domain (git adapter, signal adapter, tmux adapter, PR adapter).
That framing was vague about how the adapters plugged into the existing
protocols. This plan instead organizes work around the protocol implementations
themselves, with infrastructure modules as supporting internals.

The natural layers are:

1. **Infrastructure primitives** — thin subprocess wrappers for git, tmux, and
   gh operations. No agentrelay domain types — just `Path`, `str`, and
   subprocess calls. These are internal implementation details, not public API.

2. **Protocol implementations** — classes that satisfy `TaskPreparer`,
   `TaskLauncher`, etc. by composing infrastructure primitives and operating on
   `TaskRuntime` / `WorkstreamRuntime`. This is where the design connects.

3. **Instruction builders** — per-role text generators that produce the
   `instructions.md` content the `TaskPreparer` writes to the signal directory.

4. **Composition + CLI** — a factory that assembles `TaskRunnerIO` /
   `WorkstreamRunnerIO` from the concrete implementations, and a CLI entry
   point that wires `Orchestrator` + `TaskRunner` + `TaskGraphBuilder`.

## PR plan

### PR J: Infrastructure primitives

- Branch: `feat/infra-primitives`
- New package: `src/agentrelay/infra/`

Thin, stateless subprocess wrappers. No domain types — just paths and strings.
These are the building blocks that protocol implementations will compose.

**Modules:**

| Module | Functions | Used by protocols |
|---|---|---|
| `git.py` | `worktree_add()`, `worktree_remove()`, `branch_create()`, `branch_delete()`, `pull_ff_only()`, `fetch_branch()`, `update_local_ref()`, `push_branch()`, `ls_remote_branch_exists()` | TaskPreparer, TaskTeardown, TaskMerger, WorkstreamPreparer, WorkstreamMerger, WorkstreamTeardown |
| `tmux.py` | `new_window()`, `send_keys()`, `capture_pane()`, `kill_window()`, `wait_for_tui_ready()` | TaskLauncher, TaskKickoff, TaskTeardown |
| `gh.py` | `pr_merge()`, `pr_create()`, `pr_body()` | TaskMerger, WorkstreamMerger |
| `signals.py` | `ensure_signal_dir()`, `write_json()`, `write_text()`, `read_signal_file()`, `poll_signal_files()` | TaskPreparer, TaskCompletionChecker, TaskTeardown |

**Design notes:**
- Functions in `infra/` are pure wrappers: they take paths/strings and call
  subprocess. They raise `subprocess.CalledProcessError` on failure — the
  protocol implementations handle error mapping.
- `infra/` is a private implementation detail (prefixed with `_` in imports
  or excluded from `__init__.py` re-exports). It is not part of the public API
  and does not appear in the design diagram as a first-class package.
- The `<<module>>` stereotype convention from `CLAUDE.md` applies here.

**Acceptance criteria:**
- [ ] Each function tested against temp git repo fixtures (real git, not mocked).
- [ ] tmux functions tested via subprocess mock (no live tmux session required).
- [ ] `gh` functions tested via subprocess mock.
- [ ] Signal functions tested against temp directories.
- [ ] `pixi run check` passes.

**Acceptance criteria:**

### PR L: Task-level protocol implementations

- Branch: `feat/task-protocol-impls`
- New package: `src/agentrelay/execution/` (or similar)

Concrete classes implementing all six `TaskRunnerIO` protocols. Each class
composes infrastructure primitives from `infra/` and instruction builders from
PR K.

**Classes and the protocols they satisfy:**

| Class | Satisfies | Key operations |
|---|---|---|
| `WorktreeTaskPreparer` | `TaskPreparer` | `git.worktree_add()`, `signals.ensure_signal_dir()`, `signals.write_json()` (task_context), `signals.write_text()` (instructions.md via builder) |
| `TmuxTaskLauncher` | `TaskLauncher` | `tmux.new_window()`, launches Claude Code, returns `TmuxAgent` |
| `TmuxTaskKickoff` | `TaskKickoff` | Delegates to `agent.send_kickoff()` (which uses `tmux.wait_for_tui_ready()` + `tmux.send_keys()`) |
| `SignalCompletionChecker` | `TaskCompletionChecker` | `signals.poll_signal_files()`, parses `.done`/`.failed` into `TaskCompletionSignal` |
| `GhTaskMerger` | `TaskMerger` | `gh.pr_merge()`, `git.pull_ff_only()` or `git.fetch_branch()` + `git.update_local_ref()` |
| `WorktreeTaskTeardown` | `TaskTeardown` | `tmux.capture_pane()` (agent log), `tmux.kill_window()`, `git.worktree_remove()`, `git.branch_delete()` |

**Also in this PR: complete the `TmuxAgent` stubs.**

`TmuxAgent.from_config()` and `TmuxAgent.send_kickoff()` are currently
`NotImplementedError` stubs. This PR implements them using `infra/tmux.py`
functions. The `TaskLauncher` and `TaskKickoff` protocol implementations
delegate to `TmuxAgent` methods.

**Acceptance criteria:**
- [ ] Each class satisfies its protocol (verified by `isinstance` checks with
      `runtime_checkable` protocols).
- [ ] Unit tests with mocked `infra/` functions cover success and failure paths.
- [ ] Error mapping: `subprocess.CalledProcessError` from infra functions is
      wrapped in appropriate `IntegrationError` subclasses.
- [ ] `TaskRunnerIO` can be constructed from all six implementations.
- [ ] `pixi run check` passes.

### PR N: Composition and CLI entry point

- Branch: `feat/run-graph-cli`
- New module: `src/agentrelay/run_graph.py`

This PR wires everything together.

**Scope:**
1. Factory function(s) that build `TaskRunnerIO` and `WorkstreamRunnerIO` from
   concrete implementations, given a repo root path and tmux session name.
2. Build `TaskRunner(io=...)` and `WorkstreamRunner(io=...)`.
3. Build `Orchestrator(graph=..., task_runner=..., config=...)`.
4. CLI via `python -m agentrelay.run_graph <graph.yaml>` with flags:
   - `--max-concurrency` (default 1)
   - `--max-task-attempts` (default 1)
   - `--teardown-mode` (always / never / on_success)
   - `--tmux-session` (default from YAML or "agentrelay")
   - `--model` (override graph-level default model)
   - `--dry-run` (validate graph + build runtimes, don't execute)

**Acceptance criteria:**
- [ ] `python -m agentrelay.run_graph graphs/demo.yaml --dry-run` validates and
      prints the execution plan.
- [ ] Integration test using temp git repo + mock tmux/gh verifies the full
      wiring from graph YAML → orchestrator result.
- [ ] `pixi run check` passes.

## Open questions

1. **Package name for protocol implementations.** The plan uses `execution/`
   but alternatives include `impl/`, `adapters/`, or placing implementations
   alongside their protocol definitions (e.g. `task_runner/preparer.py`). The
   latter has the advantage of co-location but may bloat existing packages.

2. **Instruction builder scope.** The prototype's prompt construction in
   `run_graph.py` is ~300 lines with significant role-specific logic. How much
   of this should be ported vs. simplified for the first pass?

3. **WorkstreamRunner integration into Orchestrator.** The current
   `Orchestrator` does not call `WorkstreamRunner.prepare()` /
   `WorkstreamRunner.merge()` — it manages workstream state directly. Should
   PR N wire `WorkstreamRunner` into the orchestrator loop, or is that a
   separate follow-up?

4. **`infra/` testing strategy.** Git operations can be tested against real
   temp repos. Tmux and gh operations need subprocess mocks. Is there value in
   optional live-tmux integration tests (behind a marker like `@pytest.mark.
   integration`)?
