### What exists but isn't wired

- **`WorkstreamRunner`** — has `prepare()`, `merge()`, `teardown()` methods,
  but the orchestrator doesn't call them. Workstream state transitions
  (`activate`, `deactivate`, `mark_merged`, `mark_failed`) happen inline in
  the orchestrator, but no actual I/O (creating worktrees, merging integration
  PRs, tearing down worktrees) is dispatched.

- **`build_standard_runner()`** — factory that wires all six `StepDispatch`
  tables with default implementations, but nothing calls it yet.

- **Workstream implementations** (`GitWorkstreamPreparer`, `GhWorkstreamMerger`,
  `GitWorkstreamTeardown`) — concrete classes exist, not composed into anything.

### What doesn't exist yet

  line outside the prototype.
- **Reset tool** (`reset_graph.py`) — no clean architecture version.

## PR plan

### PR N1: Wire WorkstreamRunner into Orchestrator

- Branch: `feat/orchestrator-workstream-runner`

The orchestrator currently manages workstream state transitions inline but
doesn't dispatch any workstream I/O. This PR adds `WorkstreamRunner` as a
dependency of `Orchestrator` and inserts lifecycle calls at the right points
in the scheduling loop.

**Changes:**

1. Add `workstream_runner: WorkstreamRunner` field to `Orchestrator`.
2. Before first task in a workstream: call `workstream_runner.prepare(ws_runtime)`.
3. After all tasks in a workstream succeed (when `mark_merged()` fires): call
   `workstream_runner.merge(ws_runtime)`.
4. At orchestration end (or on failure): call `workstream_runner.teardown(ws_runtime)`
   for each workstream that was prepared.

The existing workstream state machine (`activate`, `deactivate`, `mark_merged`,
`mark_failed`) stays — those track scheduling state. The runner calls are the
actual I/O operations.

**Acceptance criteria:**
- [ ] Orchestrator accepts `workstream_runner` parameter.
- [ ] Workstream prepare/merge/teardown are called at correct lifecycle points.
- [ ] All existing orchestrator tests pass (with mock workstream runner).
- [ ] `pixi run check` passes.
