# Sprint Notes — 2026-03-13

> **Status: Planning.**

## Goal

Same end goal as the 2026-03-12 sprint — make the current architecture
runnable end-to-end — but with PR K and onward redesigned around the agent
communication protocol defined in `docs/AGENT_COMM_PROTOCOL.md`.

## Context

The 2026-03-12 sprint shipped PR J (`ops/` infrastructure primitives) and
began work on PR K (instruction builders). During PR K development, we
identified that the instruction builders were porting the prototype's
monolithic approach — mixing role-specific work with git commands, signaling
logic, and framework-specific actions in a single text blob.

We paused to design a clean communication protocol from first principles
(PR #91, `docs/AGENT_COMM_PROTOCOL.md`). The protocol defines five layers:

1. **Task manifest** (`manifest.json`) — structured facts about the task
2. **Work instructions** (`instructions.md`) — role-specific work only,
   template-driven for formulaic roles
3. **Workflow policies** (`policies.json`) — composable JSON, abstract
   vocabulary that adapters translate
4. **Signaling contract** — abstract signal definitions (done, failed,
   concern, gate_attempt)
5. **Framework adapter** — maps onto existing `TaskRunnerIO` protocol
   implementations

This sprint picks up from where 2026-03-12 left off, with the protocol as
the foundation for all remaining work.

## PR plan

### PR K: Protocol schemas, builders, and templates

- Branch: `feat/protocol-schemas`

Framework-agnostic data types and builders for protocol layers 1-3. No shell
commands, no framework-specific data anywhere in this PR.

**Modules:**

| Module | Responsibility |
|---|---|
| `protocol/manifest.py` | `TaskManifest` dataclass + `build_manifest(task, runtime) → dict` |
| `protocol/policies.py` | `WorkflowPolicies` dataclass + `build_policies(task, runtime) → dict` |
| `protocol/templates.py` | `resolve_instructions(role, manifest, adapter_name?) → str` — loads and parameterizes role templates |
| `spec/` package | `SpecRepresentation` protocol + `PythonStubSpec` first impl |

**Templates directory:** `src/agentrelay/templates/`

Shared role templates as markdown files:

| File | Role |
|---|---|
| `spec_writer.md` | Creative spec writing — parameterized by spec representation type |
| `test_writer.md` | Write tests against specs — parameterized by paths |
| `test_reviewer.md` | Review tests — parameterized by paths |
| `implementer.md` | Implement code, pass tests — parameterized by paths + gate |

Adapter-specific overrides go in `templates/<adapter_name>/` (e.g.,
`templates/claude_code/test_writer.md`). Resolution order:
adapter-specific > shared > per-task custom.

**Design notes:**
- Manifest builder extracts data from `Task` + `TaskRuntime` and produces
  the JSON structure defined in `AGENT_COMM_PROTOCOL.md` Layer 1.
- Policies builder reads `Task.completion_gate`, `Task.review`, etc. and
  produces the composable JSON structure from Layer 3.
- Template resolver uses `Task.role` to select template, substitutes
  `{src_paths}`, `{test_paths}`, etc. from manifest data. Falls back to
  per-task `instructions.md` for `generic` role.
- `SpecRepresentation` protocol defines how spec formats are described to
  agents. `PythonStubSpec` = Python files with docstrings and
  `raise NotImplementedError`.

**Acceptance criteria:**
- [ ] Manifest builder produces valid JSON matching the protocol schema.
- [ ] Policies builder produces valid JSON matching the protocol schema.
- [ ] Template resolver loads templates, substitutes parameters correctly.
- [ ] `generic` role falls back to custom instructions.
- [ ] `SpecRepresentation` protocol and `PythonStubSpec` are defined.
- [ ] `pixi run check` passes.
