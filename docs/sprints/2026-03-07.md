## Your specific questions

- `TaskGraph` next?  
  Yes. Implement it now.

- YAML -> runtime: `TaskGraphBuilder` or just `TaskBuilder`?  
  Use both layers:
  1. `TaskGraphBuilder`: YAML -> validated `TaskGraph` (IDs, deps, cycle checks, defaults).
  2. `TaskRuntimeBuilder` (or factory): `TaskGraph` -> `dict[str, TaskRuntime]`.

- Should `Task` run itself?  
  No. Keep `Task` immutable spec-only ([task.py](/data/git/agentrelay/main/src/agentrelay/task.py:145)). Use `TaskRunner` for execution.

- Implement `Orchestrator`?  
  Yes, but as a thin scheduler over `TaskRuntime`; delegate per-task lifecycle to `TaskRunner`.

## Original recommended implementation order (non-prototype, reference)

1. `task_graph.py`: `TaskGraph` + validation/query APIs (`ready`, `dependents`, topo/cycle checks).
2. `task_graph/builder.py`: YAML schema + `TaskGraphBuilder.from_yaml(...)`.
3. `task_runtime/builder.py`: initialize `TaskRuntime` objects from graph.
4. `task_runner.py`: one-task lifecycle state machine (`PENDING -> RUNNING -> PR_CREATED -> PR_MERGED/FAILED`).
5. `orchestrator.py`: async dispatch loop + dependency scheduling + resume hooks.
6. Adapter layer for side effects (git/worktree, signals, PR, agent launch), then wire `TmuxAgent` real implementation.
