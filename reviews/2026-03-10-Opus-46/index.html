    Architecture Review — 2026-03-10 (Opus 4.6)

    High-level review of the non-prototype architecture prior to implementing concrete adapter PRs (I–Q from sprint 2026-03-09).

    What's working well

    1. Immutable spec / mutable runtime split

    The strongest design decision in the codebase. Task, TaskGraph, WorkstreamSpec are all frozen; TaskRuntime, WorkstreamRuntime are mutable envelopes. This makes the data flow easy to reason about — specs flow down, state mutates in place, and the boundary between "what to do" and "what's happening" is always clear.

    2. The TaskRunner state machine is clean

    The lifecycle in task_runner.py is linear and explicit: PENDING → RUNNING → PR_CREATED → PR_MERGED, with FAILED as a sink. The ALLOWED_TASK_TRANSITIONS mapping makes legal edges auditable. The try/except structure around each IO call is repetitive but correct — every failure path records the error and transitions to FAILED deterministically.

    3. TaskGraph validates eagerly at construction

    No invalid graph can exist. Cycle detection, workstream hierarchy checks, dependency existence, key-ID consistency — all enforced in __init__. The MappingProxyType wrapping prevents external mutation. This is the right approach for a DAG that gets built once and queried many times.

    4. Clean protocol boundaries

    TaskRunnerIO and the integration_contracts protocols are well-factored. The four adapter protocols (workspace, signal, PR, agent launcher) map to real operational boundaries. The IntegrationAdapters bundle is a natural composition point.

    Concerns and observations

    1. Two parallel protocol layers for the same boundary

    TaskRunnerIO (in task_runner.py:69) and the four integration_contracts protocols (WorkspaceAdapter, SignalAdapter, PullRequestAdapter, AgentLauncher) describe overlapping concerns. The sprint plan says PR N will compose adapters into a concrete TaskRunnerIO, but the shape mismatch is notable:

    • TaskRunnerIO.prepare() takes only TaskRuntime; WorkspaceAdapter.ensure_workspace() takes both TaskRuntime and WorkstreamRuntime.
    • TaskRunnerIO has no concept of WorkstreamRuntime at all — it's a single-task interface. But workspace provisioning inherently needs workstream context (branch strategy, shared worktree).
    • integration_contracts.CompletionSignal has a concerns field; task_runner.TaskCompletionSignal does not.

    This will require either an adapter/translation layer that papers over the differences, or a refactor to align the two. Worth deciding now before concrete implementations exist.

    2. Task.dependencies stores Task objects, not IDs

    task.py:180 — dependencies: tuple["Task", ...]. This means every Task carries references to its full dependency tree. The builder must construct tasks in topological order and thread object references through. TaskGraph then also stores _dependency_ids as a separate index built by walking task.dependencies and extracting IDs. This creates two representations of the same information. The YAML builder already works in ID space and converts to objects at the end. If Task.dependencies were tuple[str, ...] (just IDs), the builder and graph construction would simplify. The tradeoff is that task.dependencies gives you direct object access without a graph lookup — but in practice, every caller that needs dependency info already has a TaskGraph handle.

    3. Agent on TaskRuntime is an odd fit

    task_runtime/runtime.py:113 — agent: Optional[Agent] = None. The runtime envelope bundles immutable spec + mutable state + artifacts + a live process handle. The agent reference is set by TaskRunner after launch (task_runner.py:219), but it's only used within that same run() call scope. Nothing outside TaskRunner.run() reads runtime.agent. Storing a live process handle on a data container that gets passed around and returned in OrchestratorResult is a latent lifecycle hazard — if someone holds a reference to the result after teardown, runtime.agent points to a dead process.

    4. The orchestrator does a lot of direct state mutation

    orchestrator.py directly mutates TaskRuntime.state.status, TaskRuntime.state.error, WorkstreamRuntime.state.* in multiple places — the run() method, _mark_blocked_pending_tasks_failed(), _mark_inflight_tasks_failed(), _refresh_workstream_terminal_states(). The TaskRunner also mutates the same runtime. This dual-writer pattern is manageable now because the orchestrator only touches runtimes that aren't currently in a TaskRunner.run() call, but that invariant isn't enforced — it's a convention. Something like retry (resetting FAILED → PENDING at orchestrator.py:318) directly overwrites status that the TaskRunner just set.

    5. integration_errors is elaborate but not yet consumed

    The error hierarchy (7 exception classes + classifier function) is well- structured, but nothing in task_runner.py or orchestrator.py catches IntegrationError or calls classify_integration_error(). The TaskRunner catches bare Exception at every IO boundary and does f"{type(exc).__name__}: {exc}" — no classification, no boundary tagging. The integration errors are designed to be thrown by concrete adapters and caught by... something. But the current TaskRunner doesn't distinguish expected-task-failure from internal-error at the IO level. This gap needs to be resolved when wiring concrete adapters — either the TaskRunner needs to use classify_integration_error(), or the distinction only matters at the orchestrator level (which already has TaskOutcomeClass).

    6. RemoteWorkspaceRef is speculative

    integration_contracts/__init__.py:35-54 has 6 optional fields for a remote execution model that doesn't exist yet and may never look like this. workspace_uri, repo_ref, execution_target, artifacts_uri are placeholder guesses. This is the one place the codebase speculates about future needs rather than designing for today's requirements.

    7. Workstream hierarchy depth — is it needed?

    max_workstream_depth defaults to 1 (parent→child only). The validation machinery for acyclic hierarchies, depth limits, and ancestor-chain walking in the orchestrator is thorough. But the sprint plan and existing YAML examples only show flat workstreams. The parent/child hierarchy adds ~100 lines of validation + orchestrator logic for a feature with no concrete use case yet. Not necessarily wrong — it may reflect a real need — but worth flagging.

    8. The _transition_to_failed escape hatch

    task_runner.py:279-286 has a fallback that force-sets status = FAILED even when FAILED isn't in ALLOWED_TASK_TRANSITIONS[current_status]. This bypasses the transition table for cases where the current status is a terminal that shouldn't reach FAILED (like PR_MERGED). It's defensive, but it means the transition table isn't actually authoritative — there's a backdoor. In practice this path can't be hit from _record_io_failure() because IO failures only happen during RUNNING, but the code doesn't make that guarantee visible.

    9. No logging or observability hooks

    OrchestratorEvent is a good structured-event model, but events are only accumulated in a list and returned at the end. There's no callback/listener/logging hook for real-time observability during a run. For debugging a multi-agent orchestration, you'd want to see events as they happen, not just in a post-mortem result. This is likely fine for now but will matter immediately when running real workloads.

    Summary of risk areas for the adapter PRs ahead

    The biggest design tension is item 1 — how TaskRunnerIO (single-task, no workstream awareness) composes with integration_contracts (workstream-aware adapters). This needs a clear resolution before PR N. The other items are lower risk — things to clean up opportunistically as you go.
