    Review Findings (ordered by severity)

    1. Critical: Tasks can be marked DONE without any merge if .done lacks a PR URL.
      In _run_task, task.state.status = DONE is set even when pr_url is empty, so no merge or .merged signal occurs. This can unblock dependents with missing code.
      Reference: run_graph.py:747, run_graph.py:779, run_graph.py:829

    2. Critical: If graph branch refresh fails after merge, downstream tasks can still start from stale code.
      pull_graph_branch() failure only logs a warning, but task is still marked DONE. Since new worktrees are based on local graph/<name>, dependents may run from outdated branch state.
      Reference: run_graph.py:820, run_graph.py:829, task_launcher.py:78

    3. High: create_graph_branch() can hard-reset working tree unexpectedly.
      Fallback path runs git reset --hard <base_branch> if git branch -f fails, which can discard local changes in target repo.
      Reference: task_launcher.py:50, task_launcher.py:57

    4. High: Resume behavior for .done is not implemented (doc/code mismatch).
      Docs say .done without .merged should attempt merge on startup; code only hydrates from .merged/.failed.
      Reference: agent_task_graph.py:80, WORKFLOW_DESCRIPTION.md:60

    5. Medium: Dependency graph validation is incomplete.
      _topo_sort doesn’t detect cycles or unknown deps explicitly, which can produce silent/incomplete scheduling or unclear errors.
      Reference: agent_task_graph.py:90

    6. Medium: Packaging metadata is incomplete for runtime install outside Pixi.
      pyproject.toml has empty dependencies, but runtime imports yaml.
      Reference: pyproject.toml:3, agent_task_graph.py:7

    7. Medium (architecture): Assistant backend is tightly coupled to Claude+tmux.
      claude, --dangerously-skip-permissions, tmux lifecycle, and Claude-TUI readiness checks are hardcoded, making Open/Closed and DIP weak for backend expansion (Copilot/Codex/Gemini).
      Reference: task_launcher.py:176, task_launcher.py:183, task_launcher.py:719, docs/BACKLOG.md:32

    SOLID-oriented Cleanup Opportunities (for pluggable assistants)

    1. Split orchestration into ports/adapters: AgentHarness, RepoOps, PullRequestOps, SignalStore, InstructionBuilder.
    2. Keep current behavior behind adapters first: ClaudeTmuxHarness, GitCliRepoOps, GhCliPullRequestOps, FileSignalStore.
    3. Add backend selection at graph/task level: default backend at graph level; task override for routing.
    4. Make orchestrator state explicit: use durable run_state.json with states like WAITING_AGENT, WAITING_MERGE, DONE.
    5. Refactor task_launcher.py (1105 LOC) by responsibility: split into git_ops.py, gh_ops.py, tmux_harness.py, signals.py.
    6. Add contract tests for each port and one fake end-to-end orchestrator test.
